We have developed a PIM System wherein Tenant creates their product catalouge in PIM as a single source of truth.
We can create
1.) Categories
2.) Attributes
3.) LOVs (List of Values) or Reference Values or Reference Master Data
4.) Products

Attributes are assigned to Categories
Attributes can be of following data types
1. varchar
2. float
3. int

Attributes can be mainly of 2 types
1. textBox
2. dropdown
Attributes of dropdown type will have the LOVs attached to it

Tenant can push the catalouge created in PIM to various channels (like Amazon, Flipkart. Myntra etc) integrated wit PIM
So to convert the data from PIM to Channel we have created a Mapping System wherein the Tenant can map
1.) Categories
2.) Attributes
3.) LOVs 
of PIM to that of Channels

To Achieve this we have created all the 
Categories
Attributes
LOVs of Channels in PIM in Channel Tables

To prevent repetitive work we have also created "Core" which contains all common 
Channels, Attributes, LOVs of Multiple Channel as an intermediary Entity for Mapping

What we do is
Step 1: Map Tenant's Categories, Attributes, LOVs to Core's Categories, Attributes, LOVs (Tenant Wise)
Step 2: Map Core's Categories, Attributes, LOVs to Channel's Categories, Attributes, LOVs (Channel Wise)


Tenant is Uniquely Identified by composite of tenant_id, org_id
Channel is Uniquely Identified by channel_id

Core Tables
CREATE TABLE IF NOT EXISTS public.core_categories
(
    id integer NOT NULL DEFAULT nextval('core_categories_id_seq'::regclass),
    category_name character varying COLLATE pg_catalog."default" NOT NULL,
    parent_id integer,
    depth integer,
    is_leaf boolean,
    category_path character varying COLLATE pg_catalog."default",
    CONSTRAINT core_categories_pkey PRIMARY KEY (id),
    CONSTRAINT core_categories_parent_id_fkey FOREIGN KEY (parent_id)
        REFERENCES public.core_categories (id) MATCH SIMPLE
        ON UPDATE CASCADE
        ON DELETE CASCADE
)

CREATE TABLE IF NOT EXISTS public.core_attributes
(
    id integer NOT NULL DEFAULT nextval('core_attributes_id_seq'::regclass),
    attribute_name text COLLATE pg_catalog."default" NOT NULL,
    attribute_type text COLLATE pg_catalog."default" NOT NULL,
    attribute_data_type text COLLATE pg_catalog."default" NOT NULL,
    length integer NOT NULL DEFAULT 100,
    mandatory boolean NOT NULL,
    "constraint" boolean,
    status boolean NOT NULL,
    CONSTRAINT core_attributes_pkey PRIMARY KEY (id)
)

CREATE TABLE IF NOT EXISTS public.core_category_attribute_assignment
(
    id integer NOT NULL DEFAULT nextval('core_category_assignment_id_seq'::regclass),
    category_id integer NOT NULL,
    attribute_id integer NOT NULL,
    mandatory boolean DEFAULT true,
    status boolean DEFAULT true,
    CONSTRAINT core_category_assignment_pkey PRIMARY KEY (id),
    CONSTRAINT fk_attribute FOREIGN KEY (attribute_id)

        ON UPDATE NO ACTION
        ON DELETE NO ACTION,
    CONSTRAINT fk_category FOREIGN KEY (category_id)

        ON UPDATE NO ACTION
        ON DELETE NO ACTION
)

Mapping Safeguards
Tenant -> Core, Many to One Mapping should not be possible in any Case of Categories, Attributes or LOV Mappings for a particular tenant, but vice versa can be possible
Core -> Channel, Many to One Mapping should not be possible in any Case of Categories, Attributes or LOV Mappings for a particular channel, but vice versa can be possible

Category Mapping Page
1.) We can see Tenant -> Core -> Channel Category Mappings in Single page.
2.) Tenant Category is Primary Column
2.) We can edit the Category mappings
3.) We can have filters on Tenant (Single) and Channel (Single)

Attribute Mapping Page
1.) We can see Channel -> Core -> Tenant Attribute Mappings in Single page.
2.) Channel Attributes is Primary column
3.) We can edit the Attribute mappings
4.) We can have filters on Tenant (Single), Channel (Single), Channel Category (Single), Tenant Category (Multi Select)
In Filters of Attribute Mapping
User should first select Channel Category, then in tenant category filters, only those tenant categories should come that are mapped to the selected channel category
Now for multiple Categories of a single tenant we can have same attributes and even same mappings for Channel -> Core -> Tenant
So when multiple categories are selected, we should be able to see mappings in Groups
1.) Common mappings i.e. Mappings that are common for Channel and Tenant Categories
2.) Mappings that are not common should be shown in groups of Channel Category - Tenant Categories
When we edit Common mappings, it should be reflected for all categories  

LOV Mapping Page
1.) We can see Tenant -> Core -> Channel LOV Mappings in Single page.
2.) Tenant LOVs are the primary column
3.) We can edit the LOV mappings
4.) We can have filters on Tenant (Single), Channel (Single), Channel Category (Single), Tenant Categories (Multiple), Channel Attribute (Single), Tenant Attribute (Multiple)
In the Filters of LOV Mapping

User First Selects Tenant, Channel, Channel Category
Then in Tenant category filter, only those Tenant categories should come that are mapped to the selected channel category
User Then Selects Channel Attribute
After that in Tenant Attribute Filter, only those Tenant Attributes that are mapped to the selected Channel Attribute for those Tenant <-> Channel Category Mapping
Now for a single Channel LOV, Attribute, Category Combination, multiple Tenant LOV, Attribute, Category can be mapped
So according to Filters Selected, we should be able to see mappings in Groups
